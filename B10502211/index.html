eax 是"累加器"(accumulator), 用來儲存計算產生的中間結果

ebx 是"基底暫存器"(base register),存放 邏輯位址轉換成實體位址的基底值。

項目二

Operating system interfaces

Os的工作就是讓多個程式共享計算機資源，並提供一系列比硬體更有用的服務。
os管理抽象化底層硬體，在應用程式時，不須在乎使用了哪些硬體。
Os提供程式受控的互動介面，使他們可以共享資料或共同運行
Os的管理可分為:

  1. Process
  2. i/o
  3. memory manage
  4. file system

正在運行得program叫做process，當process需要kernel service時，會發起system call，從user space換至kernel space

注意:shell是一種user program，不屬於kernel的部分，也就是說shell可以被替換。

Processes and memory

xv6的process由兩部分組成:

  1. user space的memory(包含instructions, data, stack)
  2. 只有kernel看得到每個process的狀態

Xv6是分時系統，決定哪一個ready的程式，能獲得cpu的使用權。當process不再執行時，xv6會先儲存cpu上register的值，以便下次要執行時可以恢復。

一個process可以產生新的process，藉由fork指令(system call)，新的process叫作child，原本的process叫作parent，child memory的內容與parent相同。在parent端，fork會回傳child的pid；在child端，只會回傳’0’。

執行fork後，總共會有兩個程式，一個是parent，另一個是child。child端會打印出”: child: exiting”，就結束；parent端，會等待child執行完畢，當child結束後，wait()會回傳child的pid，此時parent才可以繼續執行後續的指令。

雖然child和parent有相同的內容，但他們分別是在不同的記憶體空間，所以改變 變數，並不會影響到另一個process。

使用exec (system call)可以載入新的file，替換process原先 memory的內容。

Exec有兩個參數:

  1. 可執行檔案的名稱
  2. 字串參數矩陣

I/O and File descriptors

ﬁle descriptor是一個小整數，方便kernel管理process的讀寫。每個process都有一個table，Xv6 kernel利用ﬁle descriptor當作table的索引。每個process都有一個從0開始的ﬁle descriptor空間。

a process reads from ﬁle descriptor 0 (standard input).
writes output to ﬁle descriptor 1 (standard output).
writes error messages to ﬁle descriptor 2 (standard error).

read指令(system call):

read(fd, buf, n)，從ﬁle descriptor fd讀n bytes 到buffer，並回傳讀了多少bytes。若讀不到東西，則回傳’0’，代表文件結束。

Write指令(system call):

write(fd, buf, n)，從buffer捉n bytes寫到ﬁle descriptor fd，並回傳讀寫了多少bytes。 若回傳值小於n bytes 代表發生錯誤。

Pipes

Pipe提供process之間傳輸的方式，一端是write，另一端是read，且只能單向傳輸。

Parent要傳訊息給child:

Parent端: 把read關閉(p[0])，把訊息用write()寫出，寫完後write關閉(p[1])。
Child端: 將fd[0]的standard input改成p[0]的read，之後因為p[0]已複製到fd[0]，所以可以關閉；p[1]不會用到，也將其關閉。

Operating system organization

Os必須滿足三項要求:

  1. 多工:讓所有的processes都能被執行到。
  2. 獨立:若某個process fail，並不會影響其他process。
  3. 互動:processes之間能互相溝通，e.g. pipelines。

為確保獨立性，禁止應用程式直接存取硬體，必須間接透過system call，system call把硬體抽象化了。

每個process都有兩個stack:

  1. user stack
  2. kernel stack

當process正在執行user instruction時，只有user stack被使用，kernel stack是空的；當process進到kernel(e.g. system call)時，kernel code會在kernel stack上執行，此時user stack可能有data，但不會被執行。

當process呼叫system call時，cpu換到kernel stack，並提升硬體的特權級，再開始執行kernel instructions。System call完成後，換回user space，降低硬體的特權級，cpu換到user stack，繼續執行user instructions。

項目三

Traps, interrupts, and drivers

發生例外時，就無法繼續執行user program，控制權交給kernel，由kernel來處理。例外事件可能是: device發出信號、user program的非法操作等。
處理例外要面臨三大挑戰:

  1. kernel必須使cpu從user mode換至kernel mode
  2. kernel和device必須協調好它們平行的操作
  3. kernel必須了解device的介面

Systems calls, exceptions, and interrupts

Systems call: user program請求os服務
Exception:程式執行到非法的操作(e.g.除以0、試圖存取不存在PTE的內存)
Interrupt:device發出信號，提醒os注意(e.g. i/o完成、時脈晶片每100ms產生interrupt，使kernel可以實現time sharing)

所有的中斷都由kernel管理，而不是process。因為在大多數情況下只有kernel擁有處理中斷所需的特權和狀態。

當發生Interrupt，cpu暫停正在執行的程式，去執行interrupt handler，但在執行interrupt handler之前，cpu會先儲存registers內的值，以便interrupt處理完後，os可以恢復該程式上次執行的狀態。

術語定義:

Trap: 由軟體(process)產生的。
Interrupt:由硬體(device)產生的，與正在執行的process無關。

X86 protection

x86有四個特權級，從0(特權最高）到 3（特權最低）。當下的特權級是存在%cs register。

interrupt handler由interrupt descriptor table (IDT)定義，IDT有256個項目，每項提供相對應的%cs 和 %eip。
